## Afternoon Scan — April 19, 2026 (Run 2)

Morning run covered micro-lesson, pantry snapshot, and weekly-plan re-eval. This second scan looks further down the Section 11 bench and picks up three unclaimed levers surfaced by today's fresh web evidence on social-video imports, "Cooking Mode" hygiene, and graceful-failure retention mechanics.

1. **Social-Video URL → Closest-Catalog Match in TextPrompt** (NEW Section 11 candidate)
   - What: Paste a TikTok / Instagram / YouTube cooking URL into the existing Today TextPrompt. Regex detects URL → fetch oEmbed title + description → feed the text straight into the already-built `find-closest-dishes` semantic matcher → return the closest existing Sous catalog dish with a single "Why this match?" line. Zero new screen, zero new UI element — the input is the same TextPrompt, the output is the same results sheet.
   - Why now: [Honeydew explicitly pitches "import from Instagram, TikTok, Pinterest, YouTube"](https://honeydewcook.com/blog/recipe-apps-social-media-imports) as its marquee 2026 feature, and the broader [short-video-first food discovery shift](https://tastewise.io/blog/social-media-food-trends) is the defining cooking-app behavior change of this cycle. Every competitor is racing to become a URL scraper. Sous's contrarian move — governed by Rule #7 "no invented recipes or images" — is to use the URL as a craving signal, not a recipe source: take the video's title/description, semantically match to the existing catalog, and run the proven guided cook. Captures the entire "I saw this on TikTok" entry loop without drifting off-catalog.
   - Effort: Low — URL detection in `text-prompt.tsx`, one `/api/oembed` proxy route (oEmbed is a two-line JSON fetch), reuse `find-closest-dishes.ts` unchanged.
   - Impact on cooks-per-week: High for the 18–35 cohort, where short-video is the dominant recipe-discovery surface. Converts passive scroll into a committed cook without ever leaving Sous's catalog.
   - Recommendation: **Build** — passes every Section 11 gate, respects Rule #7 exactly, and plants a flag on the biggest 2026 discovery shift without becoming a URL-scraper product.

2. **Wake Lock API — Silent Screen-Awake During Active Cook** (Section 11 — zero UI)
   - What: Acquire a `navigator.wakeLock.request('screen')` when the user enters the `/cook/[slug]` Cook phase. Release on Win, Pause, route change, or tab background. No icon, no toggle, no indicator — completely invisible.
   - Why now: "Cooking Mode" (screen-awake + large text) is the table-stakes hygiene feature every competitor now ships — [Preplo](https://preplo.app/best-recipe-app-2026), [Flavorish](https://www.flavorish.ai/blog/cooking-mode-on-flavorish), [Feast by The Guardian](https://preplo.app/best-recipe-app-2026), and [Drizzle Lemons](https://www.drizzlelemons.com/blog/cook-mode-step-by-step-recipe-view) all call it out as a 2026 standard. Sous already has readable typography (NEXT-20 P1) and big-hands mode (NEXT-20 P13), so the screen-awake piece is the one missing primitive of the standard. Mid-step auto-lock is an invisible churn vector — the user taps with a wet hand, fails, frustration compounds.
   - Effort: Low — ~20 lines in a `useWakeLock()` hook with cleanup on unmount + visibility-change listener. Graceful fallback for Safari versions without the API.
   - Impact on cooks-per-week: Medium. Doesn't add new behavior, but removes a sharp failure mode during the exact moment (mid-recipe) the user is most committed. Compounds with read-aloud and `IngredientList` station toggle — reduces the total friction surface of the Cook phase.
   - Recommendation: **Build** — the cheapest infrastructure upgrade on the board, zero UI, catches us up to the 2026 Cooking-Mode baseline every competitor has already shipped.

3. **"Welcome Back from a Gap" Copy Variant on Today** (Section 11 — content variant)
   - What: Current Today welcome line (NEXT-20D P4) covers streaks of 0 / 1–2 / 3+ days. Add a fourth variant: when `lastCookAt` is 3–14 days ago and streak is broken, render a warm re-entry line — "Back in the kitchen? Let's do one easy thing tonight." — instead of falling through to the default streak-zero copy. Pure content variant on an existing component, no new UI element.
   - Why now: The [2026 retention research consensus](https://enable3.io/blog/app-retention-benchmarks-2025) and the [UCL habit-formation finding](https://habi.app/insights/best-streak-tracker-apps/) both converge on "graceful failure handling" as the single highest-ROI re-activation lever. [Calm and Apple Fitness](https://habi.app/insights/best-streak-tracker-apps/) don't punish lapses; they welcome returners. Sous's streak architecture is already sophisticated (earned freezes are in scope, rest-day grace shipped in NEXT-20 P16), but the *linguistic* re-entry still treats a 7-day-gap user and a day-one user identically. The emotional valence of "you broke your streak" vs. "back in the kitchen?" is the difference between the user opening the app twice more this month and not.
   - Effort: Very Low — one new branch in `deriveWelcomeCopy` + one additional test case. No new data, no new UI.
   - Impact on cooks-per-week: Medium-high over a 90-day window. Lapsed-user reactivation is the hardest retention cohort; softening the re-entry line is the cheapest intervention with measurable lift in the meditation/fitness literature.
   - Recommendation: **Build** — a five-minute change that closes Sous's last sharp-edge interaction for the exact cohort (anxious, intermittent cooks) the product was designed to serve.
